### アプリの状態（state）の形
localStorage キー **`taskapp-state-v1`** に、次の形のJSONが入る:
```jsonc
{
  "projects": [
    {
      "id": "xxxx", "name": "...", "emoji": "🚀", "color": "#007aff", "notes": "...",
      "tasks": [
        { "id": "yyyy", "text": "...", "done": false, "deadline": null, "memo": "", "subtasks": [] }
      ],
      "deleted": false, "deletedAt": null   // ← プロジェクト削除は「ソフト削除」(配列から消さずフラグを立てる)
    }
  ],
  "custom": [], "recent": [], "countdowns": [], "hidden": [], "headline": ""
}
```
- **重要**: プロジェクトの「削除」「完了」は配列から消さず、**フラグ（`deleted`/`deletedAt` 等）で表現される**。マージ時にこれらのフィールドを取りこぼすと「削除/完了が元に戻る」バグになる（過去に発生・修正済み）。

## 4. 同期エンジンの設計（現行 = 作り直し後）

過去に個別バグ修正を重ねて複雑化し、**オフライン絡みで「同期OK／マージ中」が高速に無限ループ**する問題が出たため、制御部分を**1から作り直した**。マージ計算とアプリ本体は変更していない。

### 使う localStorage / sessionStorage キー
| キー | 用途 |
|------|------|
| `taskapp-state-v1` | アプリ本体が読み書きする全状態（元アプリ仕様） |
| `taskapp-base` | 3-wayマージの「前回同期した共通の土台（ancestor）」 |
| `taskapp-backups` | 上書き前の自動バックアップ（直近8件、保険。UIは無し） |
| `taskapp-sync-rev` | 旧実装の名残（現行ロジックではほぼ未使用） |
| `taskapp-scroll`(session) | 自動リロード時のスクロール位置復元用 |

### 中核ロジック（`reconcile()` に一本化）
1. 変化（クラウド受信 or 手元編集）があったら `reconcile()` だけが動く。
2. クラウド購読 `onSnapshot` で `lastCloud` を更新 → `reconcile()`。
3. 手元編集は `Storage.prototype.setItem` をフックし、**600msデバウンス**で `reconcile()`。
4. `reconcile()` は「手元(local)・クラウド(cloud)・土台(base)」を見て:
   - `local === cloud` → **ここだけ `base` を更新**（＝両者が一致した瞬間のみ土台を確定）。これが無限ループ防止の肝。
   - 手付かずの初期サンプル端末 → クラウドを取り込む。
   - クラウドが空/初期化で手元が本物 → 手元で回復（自己修復）。
   - 両方に本物データ → **3-wayマージ**。`merged !== cloud` のときだけ送信、`merged !== local` のときだけ画面反映（再読込）。
5. **再読込中(`reloadPending`)・書き戻し中(`applyingRemote`)は何もしない**（再入防止）。

### マージ方針（ユーザー決定事項）
- 別々のタスクの**追加**は両方残す。片方だけの**編集/削除**は尊重。
- **同じタスクを両端末で別々に編集**した競合 → **両方を複製して残す**（決定論的IDで収束。`元ID + "~d" + ハッシュ`）。
- **削除 vs 編集** の競合 → **編集を優先**（削除を取り消す）。
- プロジェクトの**全フィールド**を3-wayマージ（`deleted`/完了フラグ等も含む）。`tasks` のみ個別にマージ。
- **出力の並び順は「並び替え対応の3-wayマージ」**（`mergeOrderIds(base,local,remote)`）。誰も並び替えていなければ従来どおり「クラウド順→手元だけの追加を末尾」で全端末同一に収束（ループしない）。**片方だけ並び替えたらその並びを採用**（＝並び替えが直後に元へ戻らない問題を修正）。両方が別々に並び替えた競合はクラウド優先で決定論的に収束。
- 競合できないスカラー（プロジェクト名・headline等）→ クラウド側優先（`pick3`）。

### オフライン対応
- **オフライン中はクラウドへ送らない**（`navigator.onLine===false` ならスキップ）。古い内容のまま復帰時に送信して他端末の変更を上書きするのを防ぐ。
- **オンライン復帰時**は `getDocFromServer` で**サーバの最新クラウドを読み直してからマージ送信**する（`online` イベント）。
- **【v6】サーバ確定値ガード（`cloudFresh`）**: 再接続直後に Firestore が出す「キャッシュ（古いクラウド値）」を土台に送信すると、相手端末の最新タスクを上書きしてしまう。これを防ぐため、`onSnapshot` を `{includeMetadataChanges:true}` で購読し、`snap.metadata.fromCache===false`（＝サーバ確定値）のときだけ `cloudFresh=true` にして同期を動かす。`offline` イベントや復帰直後のキャッシュ受信中は `cloudFresh=false` で送信/反映を止める。→ **「オフライン端末が復帰時に相手の追加タスクを消す」不具合の根本修正**。
- 手元編集は即 localStorage に保存されるため、**アプリを閉じても・再起動しても残る**（写真/メモアプリ的な永続化）。

### その他の安全機構
- **サービスワーカー** `sw.js`（network-first）: オンライン時は常に最新の `index.html` を取得。古いキャッシュ/古いタブが原因の初期化・上書きを防ぐ。オフラインでも起動可能。
- **初期サンプル保護**: 手付かずの「サンプル」状態でクラウドの本物を上書きしない（`isDefaultSample` / `isEmptyOrDefault`）。
- **復元モード**: URLに `?recover` を付けると、自動同期を止めて端末内バックアップ（`taskapp-backups`）から復元できる画面が出る（通常時は非表示。誤タップ防止）。
- 画面左下に **☁️ 同期ステータス表示**（接続中/同期OK/保存中/オフライン等）。

## 5. これまでの主な決定事項

- 完成済みHTMLは**中身を変えず** `index.html` として配置。同期は末尾追記方式。
- リポジトリは**公開**にして GitHub Pages（無料）で配信。Pages は **branch配信**（Actions不使用）。
- 同期の競合方針: 同編集=**複製**、削除vs編集=**編集優先**（ユーザー選択）。
- 「他デバイスの変更を完全に無リロードで反映」は、アプリ本体（ミニファイ済み）が外部変更を受け取る口を持たないため**不可**。代わりに「入力を邪魔せず・スクロール位置を保つ自動リロード」で近づけている（自分の操作は元から無リロード）。
- 過去のデータ消失（初期化事故）分は**復元不可として了承済み**。以後は上書き前バックアップ＋各種ガードで再発防止。

## 8. 開発・検証の進め方（メモ）

- 同期ロジックを変更したら、**Nodeでマージ関数を抽出して単体テスト＋2台シミュレーション**（収束・無限ループ無し）を回してからコミットするのが安全（過去にこの方法で多数のバグ・ループを検出）。
- `index.html` 末尾の同期ブリッジは `<!-- firestore-sync-bridge -->` を目印に置換・編集している。アプリ本体（その手前の巨大なミニファイ済みコード）は触らない。
- コミット後は `git push origin main`。GitHub Pages が自動で更新（各端末への反映は最大10分程度のキャッシュ遅延あり）。
